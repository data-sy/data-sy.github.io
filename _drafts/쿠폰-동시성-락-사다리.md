## 사건: 100장 한정인데 999장이 나갔다

- 사실: 한정수량 100짜리 선착순 쿠폰. 락 없이 동시 요청을 흘렸더니 **실제 발급 행 999개** (장부 카운터는 100). 10배 초과.
- 이게 왜 일어나는지 한 문단 (read-modify-write 경합, check-then-act).

✍️ **당신이 쓸 것:** 이 숫자를 처음 봤을 때 뭘 의심했나? "코드가 틀렸나" 아니면 "동시성이구나"가 먼저였나?
처음부터 락 5종을 계획한 게 아니라면, 어쩌다 사다리를 다 타게 됐는지 그 동기를 솔직하게.

## 측정을 먼저 신뢰할 수 있게 만들었다

- 사실: 버전 간 throughput은 **비교하지 않기로** 했다 — 경합 전략이 같이 움직여서 교란. 비교한 건 오직 **"초과를 막았나(actual ≤ total)"** 이진 사실.
- 통제변수 핀: (리포트 §3에서 실제로 고정한 변수 — VU 수, 워크로드, 쿠폰 수량 등 확인해서 본인 말로)

✍️ **당신이 쓸 것:** "수치를 비교하지 않기로 한 결정"이 이 글의 차별점이다. 왜 그 함정을 경계했는지, 처음엔 throughput을 비교하고 싶지 않았는지.

## 사다리 v1 → v5

각 버전을 한 문단씩. **자랑이 아니라 "여기서 뭐가 부족해서 다음 칸으로 갔나"** 흐름으로.

| 버전 | 메커니즘 | 초과 막음? | 한계 / 다음으로 간 이유 |
|---|---|---|---|
| v1 | 락 없음 | ✗ (999) | — |
| v2 | `synchronized` | ✓ | 단일 JVM 한정 (멀티 인스턴스에서 깨짐) |
| v3 | DB 행 락 `SELECT ... FOR UPDATE` | ✓ | — (최종 채택) |
| v4 | Redisson 분산 락 (워치독) | ✓ | (리포트 §6 기각 근거) |
| v5 | 커스텀 SET NX 락 (고정 TTL) | ✓ | fencing 공백 노출 |

✍️ **당신이 쓸 것:** 각 칸에서 *실제로 막혔던 지점*. 특히 v2→v3, v4를 만들어보고 기각한 과정.
표는 요약이고, 본문은 당신이 디버깅하며 본 것.

## fencing 스톨 데모 — "안전한 release ≠ 상호배제"

- 사실: 만료-중간탈취를 주입했더니 **해제는 거부됐는데도**(오해제 차단 ✓) `actual=2 > total=1`. fencing token 없이는 어떤 분산 락도 못 막는 공백을 **실측**으로 보였다.
- 이게 왜 중요한지: 분산 락을 "걸면 안전"이라 믿는 흔한 오해를 깬다.

✍️ **당신이 쓸 것:** 이 데모를 직접 짜서 재현한 경험. 이게 면접에서 가장 강한 카드다 — "분산 락 써봤다"가 아니라 "분산 락의 한계를 실측했다".

## 결정: Redis 분산 락을 기각하고 DB 행 락(v3)으로 한 칸 내려왔다

- 사실: 채택 근거는 **속도가 아니라 구조적 안전**. 락↔데이터가 한 트랜잭션이라 fencing 공백이 구조적으로 없다.
- 이 워크로드의 특성: 임계 구간이 곧 DB 쓰기 → 분산 락은 과한 외부 의존.

✍️ **당신이 쓸 것:** "더 고급인 분산 락을 안 쓰고 더 단순한 DB 락을 골랐다"는 게 핵심 메시지.
신입/주니어가 흔히 "최신·고급"을 고르는 것과 반대 방향의 판단을 했다는 점을 당신 언어로.
